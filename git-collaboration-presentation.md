# 🚀 组会分享纲要：Git 深度协作与团队代码管理

> **分享目标**：告别“压缩包传代码”和“混乱的提交历史”，建立实验室规范的多人协作模式，防止代码资产流失，提升整体代码质量。

---

## 一、 引言：我们为什么需要规范 Git？
*   **痛点重现**：师兄毕业代码跑不通？多人改同一个文件导致代码被覆盖？提交历史像一团乱麻看不出逻辑？
*   **核心理念**：Git 不仅是一个“带版本控制的网盘”，更是一套**社交化的工程协作规范**。

---

## 二、 核心篇 1：Git Flow 与主流分支管理模型
*(💡 核心对比：不同规模项目的生存之道)*

### 1. Git Flow (经典重型流)
*   **特点**：极其严谨，分支拓扑图呈现复杂的“菱形网格”，但心智负担较重。
*   **模型结构**：
    *   **2 条长期主干**：`master`（只存放随时可发布的稳定代码）、`develop`（日常开发大本营）。
    *   **3 种短期辅助**：`feature`（开发新功能）、`release`（发布前测试与修Bug的缓冲）、`hotfix`（紧急修复线上Bug）。
*   **适用场景**：大型传统商业软件、多版本并行维护的项目。

### 2. GitHub Flow (敏捷极简流) —— 实验室首选
*   **核心法则**：只有 1 条主干（`main`）。所有开发拉取临时分支，通过 PR 合并，**随时可发布**。
*   **合并策略**：大量使用 `Squash and Merge`（压缩合并），保持主干呈现极致的单线直线。
*   **适用场景**：实验室日常论文库、`EasyTier` 等敏捷开源项目。

### 3. GitLab Flow (环境流水线)
*   **核心法则**：代码跟随物理环境流转。`main` (开发) -> `pre-prod` (测试) -> `production` (线上)。
*   **适用场景**：需要部署到服务器的 ToB / SaaS 后台项目。

> **🌟 拓展科普：FFmpeg 的补丁驱动流**
> 像 Linux 内核、FFmpeg 等极度硬核项目，全程不用 Merge（拒接网格状分支），全靠打补丁（`git am`）或变基（`rebase`）融入主干。用 `Cherry-pick` 精确将单点修复摘取到老版本（LTS）分支。

## 三、 核心篇 2：冲突解决 (Merge Conflict) 的艺术
*   **冲突的本质**：多人并行修改了同一文件的同一行代码，Git 无法自动决定去留。
*   **案发现场**：张三在 `develop` 写新功能，李四从 `master` 拉了 `hotfix` 修 Bug，李四修完将代码合并回 `develop` 时，与张三的代码发生碰撞。
*   **解决冲突的“黄金铁律”**：
    *   **谁提的 PR，谁负责解决冲突**（导师/审查者绝不代劳）。
    *   **强制沟通机制**：遇到不懂的他人代码，必须当面/线上呼叫原作者对齐，绝不能盲目覆盖（Accept Both / Accept Current）。
*   **如何预防冲突？**：任务拆分要解耦；小步快跑勤提交；每天写代码前先 `git pull` 同步主干。

## 四、 核心篇 3：规范的 Commit Message 编写
*   **为什么重要**：Commit Message 就是代码的“航海日志”，是生成自动化 Changelog 的基础。
*   **约定式规范 (Conventional Commits)** 结构：`<类型>(<作用域>): <描述>`
    *   `feat(user): 新增头像上传功能` (新特性)
    *   `fix(auth): 修复登录接口崩溃的 Bug` (修缺陷)
    *   `refactor(ui): 重构首页按钮组件` (不加功能不修Bug的重构)
    *   `build(deps): 升级 vite 版本` (工程构建/依赖变动)
*   **最佳实践**：**原子化提交 (Atomic Commits)**。一个 Commit 只做一件逻辑完整的小事。拒绝“憋了半个月，几千行代码只用一个 commit 提交”。

## 五、 核心篇 4：Pull Request (PR) 流程与质量防线
*   **PR 是什么**：平台提供的“请审查并合并我的代码”的电子审批单。
*   **PR 的核心价值（防线机制）**：
    1.  **Code Review (代码审查)**：代码必须经过第二双眼睛。审查者重点看逻辑漏洞和**解决冲突时是否误删他人代码**。
    2.  **触发自动化 (CI/CD)**：合并前自动运行测试机器人，测试不通过锁定合并按钮。

### 真实开源项目案例剖析
*   **案例 1：EasyTier —— 无测试，不合并 (No Test, No Merge)**
    > **🔗 PR 链接**：[EasyTier PR #1980 - Fix IPv6 TCP tunnel display formatting](https://github.com/EasyTier/EasyTier/pull/1980)
    *   **背景**：修复 IPv6 隧道显示 Bug。
    *   **亮点**：开发者在 PR 中强制捆绑提交了 3 个配套单元测试。
    *   **高光**：AI 审查机器人发现测试用例未覆盖真实边缘场景，强硬打回要求补充测试。**结论：测试代码的审查优先级极高！**

*   **案例 2：AstrBot —— AI 审查拦截与原子化提交**
    > **🔗 PR 链接**：[AstrBot PR #4766 - fix(mcp): cannot rename MCP Server](https://github.com/AstrBotDevs/AstrBot/pull/4766)
    *   **背景**：修复无法重命名服务器的 Bug。
    *   **亮点**：标题严格遵循规范 (`fix(mcp): ...`)；AI 机器人敏锐发现“静默覆盖他人配置”的漏洞。
    *   **高光**：开发者听取建议后追加了**第二次 Commit**（展示了原子化提交的过程）。虽然没写自动化测试，但用强制的**人工运行截图**和 Checklist 妥协防线。

## 🎯 附录：Q&A 模拟问答环节预案

**Q1：大家都在不同的分支开发，合并时肯定会产生冲突（Conflict），这该怎么解决？**
> **回答策略**：
> 1. 冲突是多人协作的必然，不是错误。
> 2. **黄金铁律**：谁提的 PR，谁负责在本地拉取最新主干代码解决冲突！导师/Reviewer 只负责看结果，不负责修代码。
> 3. 解决冲突时遇到不懂的逻辑，必须当面/线上呼叫原作者沟通，绝不能盲目覆盖。

**Q2：万一有人在解决冲突时，把别人辛苦写好的代码或者测试用例给删了/改坏了怎么办？**
> **回答策略**：
> 现代工程有三重防线兜底：
> 1. **强制沟通**：这是团队素质的要求。
> 2. **Code Review**：审查者会重点看 `Merge Commit` 的 Diff 差异，发现乱改别人代码直接打回。
> 3. **自动化流水线 (CI)**：即便蒙混过关，只要测试用例报错，系统的红灯亮起，合并按钮就会被永久锁定。

**Q3：理论上，如果一个 PR 的提交者在本地不控制版本，最后只提交一次，那这个 PR 分支上是不是就只有一个节点？**
> **回答策略**：
> 没错！这恰恰是我们在实验室要极力避免的“网盘式” Git 用法。
> 只有一个节点的巨大 PR 是一场灾难：别人没法 Review，出错没法精确撤销（revert 整个点会把好代码也删了），写代码时电脑死机也没有中间存档。必须养成“小步提交（Atomic Commits）”的好习惯。

**Q4：修个 Bug 还要写测试，新功能的测试哪来的？不会也是别人代写的吧？**
> **回答策略**：
> 谁写业务代码，谁就必须在同一个 PR 里捆绑提交测试代码。
> 测试绝不是别人代写的，但必须经过别人的**交叉审查**。如果你的测试写得很脆弱（依赖本地路径），或者只是敷衍了事（比如把断言注释掉），在 PR 审查阶段一眼就会被识破并驳回。
