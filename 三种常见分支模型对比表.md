# 三大主流 Git 分支管理模型对比表

| 对比维度 |  Git Flow  |  GitHub Flow  |  GitLab Flow  |
| :--- | :--- | :--- | :--- |
| **长期存在的主干** | **2条** (`master`, `develop`) | **1条** (`main`/`master`) | **多条** (`main`, `pre-prod`, `production`...) |
| **日常开发起点** | 从 `develop` 拉取 `feature/*` 分支 | 从 `main` 拉取功能分支 | 从 `main` 拉取功能分支 |
| **发布(上线)方式** | 从 `develop` 创建 `release/*` 分支测试，最后合并入 `master` 和 `develop` | 功能分支(PR)通过测试后，合并入 `main` 即刻发布上线 | 将代码依次向右合并：`main` -> `pre-prod` -> `production` |
| **紧急修Bug (Hotfix)** | 从 `master` 拉出 `hotfix/*` 分支，修复后双向合并回 `master` 和 `develop` | 直接从 `main` 拉取分支，修复后作为一个普通的 PR 合并回 `main` | 从对应的环境分支（如 `production`）修复，然后向下合并（Cherry-pick）回 `main` |
| **核心设计理念** | **“严格隔离开发与发布”**：用复杂的流程换取极其严谨的版本管理。 | 主干始终保持可部署状态，通过持续集成保证质量。 | **“代码跟随物理环境流转”**：分支结构与公司的测试、生产服务器环境一一映射。 |
| **最大优点** | 体系极其严密，考虑到了软件生命周期中的各种极端情况，容错率高。 | 极致简单，心智负担小，极具敏捷性，天然契合 CI/CD 自动化部署。 | 完美解决了 GitHub Flow 无法控制发版节奏、以及多环境联调测试的问题。 |
| **最大缺点** | 流程太重太复杂，切来切去容易造成“合并地狱”（忘了双向合并）。 | 过于简单，对于非SaaS项目或有固定发版周期的企业级项目完全 hold 不住。 | 环境一多，分支维护成本也会上升，向后游合并时可能出现冲突。 |
| **典型适用场景** | 传统商业软件、发版周期极长的大型客户端、需要维护旧版本的系统。 | **开源项目（如 EasyTier）、敏捷型互联网团队。** | ToB 企业级后台开发、需要走严格测试与产品验收流程的 SaaS 服务。 |
